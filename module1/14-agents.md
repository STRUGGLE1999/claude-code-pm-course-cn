# 1.4: 用于并行工作的 Agents

- **完成时间：** 25 分钟
- **前置条件：** 完成 Module 1.3（文件操作和 @ 引用）

**在 Claude Code 中开始本模块：** 运行 `/start-1-4` 启动交互式体验。

## 📖 概述

Module 1.4 介绍 **agents** - 并行运行的独立 Claude 实例。不需要按顺序处理 10 份会议笔记（50 分钟），而是启动 10 个 agents 同时工作（5 分钟）。

**关键要点：** Agents 将 Claude Code 从单个助手转变为可扩展的团队。这不仅更快 - 还能解锁全新的工作流程。

## 🤯 什么是 Agents？

### 核心概念

**Agents 是并行运行的独立 Claude 实例。**

可以把它想象成克隆自己同时处理多个任务：

**顺序工作（没有 agents）：**

```
任务 1（5 分钟）→ 任务 2（5 分钟）→ 任务 3（5 分钟）→ ... → 任务 10（5 分钟）
总时间：50 分钟
```

**并行工作（有 agents）：**

```
任务 1（5 分钟）┐
任务 2（5 分钟）│
任务 3（5 分钟）├─→ 全部同时进行
... │
任务 10（5 分钟）┘
总时间：5 分钟
```

### Agent 能力

| 每个 agent 拥有 | 每个 agent 不拥有 |
|---|---|
| 完整的文件系统访问权限 | 与其他 agents 的通信能力 |
| Claude 的所有能力 | 共享状态 |
| 独立的上下文 | 其他 agents 结果的知识（直到综合时） |
| 工具使用（读/写文件） | 顺序依赖关系 |

## 🎯 何时使用 Agents

### 决策框架

| 场景 | 使用 Agents? | Agent 数量 |
|---|---|---|
| 批量处理（10 个会议，20 个访谈） | ✅ 是 | 多个（5-20+） |
| 并行研究（5 个竞争对手，3 个市场） | ✅ 是 | 少量（2-5） |
| 多来源分析（访谈 + 调查 + 工单） | ✅ 是 | 少量（2-4） |
| 单个任务 | ❌ 否 | 0 - 直接使用 Claude |
| 顺序工作（任务相互依赖） | ❌ 否 | 0 |
| 快速任务（几秒钟完成） | ❌ 否 | 0 |

### 使用 Agents 的场景：

- **批量处理：** 对多个项目执行相同类型的工作
- **并行研究：** 独立研究多个实体
- **多来源分析：** 对不同来源进行不同分析
- **时间敏感工作：** 需要在几小时内而不是几天内得到结果

### 跳过 Agents 的场景：

- 单个任务（直接向 Claude 提问）
- 顺序工作（相互依赖的任务）
- 快速任务（几秒钟完成的简单请求）
- 依赖上下文的工作（需要基于之前的工作进行）

## 🔧 如何为 Agents 编写提示

### 模式 1：明确的 Agent 请求

```
创建 10 个 agents 来处理 /meetings 中的 10 份会议笔记。每个 agent 应该提取：
- 关键决策
- 行动项
- 阻塞问题
- 下一步

将所有结果合并到 @weekly-summary.md
```

### 模式 2：让 Claude 决定

```
我有 15 个竞争对手网站需要研究功能和定价。
并行工作以快速完成。
```

### 模式 3：专业化 Agents

```
启动 3 个专业化 agents：
- Agent 1：分析 /interviews 中的所有访谈记录
- Agent 2：处理 @survey-results.csv 中的调查数据
- Agent 3：审查 /support 中的支持工单

将发现综合到 @user-research-insights.md
```

## 💼 真实世界示例

### 示例 1：会议处理

**场景：** 周一早上，10 份会议记录，1 小时后有站会。

**提示：**

```
我有 10 份会议记录在 /meetings/last-week/。
创建 10 个 agents 同时处理每个会议。

对于每个会议，提取：
- 做出的关键决策
- 行动项（带负责人和截止日期）
- 提出的阻塞问题或风险

将所有发现综合到 @monday-standup-prep.md，包含：
- 第 1 部分：关键阻塞问题
- 第 2 部分：关键决策
- 第 3 部分：按团队成员分组的行动项
```

**结果：** 5 分钟 vs 60 分钟的顺序工作。

### 示例 2：竞争研究

**场景：** CEO 需要在今天结束前完成 5 个竞争对手的竞争分析。

**提示：**

```
启动 5 个 agents 同时研究这些竞争对手：
- Asana, Linear, Monday.com, ClickUp, Notion

对于每个竞争对手，研究：
- 产品功能和能力
- 定价层级和打包方式
- 目标市场和定位
- 最近更新（过去 6 个月）
- 用户评价和情绪

创建单独文件：@competitor-[name]-research.md

然后综合到 @competitive-analysis.md，包含：
- 功能对比矩阵
- 定价对比
- 定位图
- 差距和机会
```

**结果：** 1 小时 vs 5.5 小时的顺序工作。

## 🎨 Agent 编排模式

### 模式 1：扇出处理

**用于：** 多个相似任务，需要合并所有结果

```
用 15 个 agents 处理 /meetings/ 中的所有 15 个文件。
从每个文件提取行动项。
合并为按负责人分组的单一行动项列表。
```

### 模式 2：专业化角色

**用于：** 对不同来源进行不同类型的分析

```
启动 3 个专业化 agents：
- Agent 1（访谈专家）：分析定性数据
- Agent 2（数据分析师）：处理调查数字
- Agent 3（支持分析师）：分类工单主题

综合为全面的用户研究报告。
```

### 模式 3：并行研究

**用于：** 独立研究多个实体

```
并行研究这 5 个竞争对手。
每个 agent 创建详细档案。
综合为对比矩阵。
```

### 模式 4：批量文档生成

**用于：** 创建多个相似文档

```
从 @prd.md 生成 20 个带验收标准的用户故事。
使用 agents 并行创建故事。
确保格式和质量一致。
```

### 模式 5：验证流水线

**用于：** 审查多个文档的质量

```
根据我们的质量检查清单审查 /docs/prds/ 中的所有 PRD。
每个 agent 审查一个 PRD。
创建问题和建议的摘要报告。
```

## 💡 最佳实践

### 应该做：

- **用于时间敏感的批量工作** - 当你需要快速得到结果时

- **让 Claude 决定 agent 数量** - "并行处理这些文件" 效果很好

- **提供清晰的单个任务说明** - 每个 agent 需要知道要做什么

- **请求综合** - 告诉 Claude 有意义地合并 agent 结果

- **指定输出格式** - 标准化 agent 输出以便更容易综合

### 不应该做：

- **不要用于单个任务** - 直接向 Claude 提问

- **不要创建需要通信的 agents** - 它们独立工作

- **不要用于顺序工作** - 有依赖关系的任务应该按顺序完成

- **不要忘记综合** - 原始 agent 输出需要合并

- **不要对小任务过度使用 agents** - 如果只需 30 秒，不要使用 agents

## 🐛 故障排除

**Agents 没有并行工作？**

- 明确说："创建 agents 并行工作" 或 "同时处理"

- 使用"同时"等词语

**Agent 结果不一致？**

- 提供清晰模板：
```
每个 agent 应该输出包含以下部分的 markdown：
## 摘要
## 关键发现
## 建议
```

**创建了太多 agents？**

- 指定确切数量："创建正好 5 个 agents"

- 或提供约束："使用所需数量的 agents，最多 10 个"

**agents 完成后没有综合？**

- 始终以："所有 agents 完成后，将结果综合到 @summary.md" 结尾

**一个 agent 失败了？**

- 检查错误消息了解具体问题

- 确保所有引用的文件存在

- 验证说明清晰
