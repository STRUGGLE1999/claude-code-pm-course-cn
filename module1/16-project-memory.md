# 1.6: 项目记忆

- **完成时间：** 20-25 分钟
- **前置条件：** 完成 Modules 1.1-1.5

**在 Claude Code 中开始本模块：** 运行 `/start-1-6` 启动交互式体验。

## 📖 概述

CLAUDE.md 是 Claude 对你项目的永久记忆。写一次，Claude 在每次对话中都知道你的产品上下文 - 无需重新解释。

**关键要点：** CLAUDE.md 是你项目的宪法 - 覆盖临时提示的不可变规则。虽然提示是灵活的请求，但 CLAUDE.md 确立了你项目的最高法则。

## 🏛️ 什么是 CLAUDE.md？

### 核心概念

CLAUDE.md 是项目目录中的 markdown 文件，包含关于你的产品、团队和偏好的永久上下文。Claude 在会话开始时自动读取它并应用其中的所有内容。

**三个关键属性：**

- **自动加载：** Claude 在会话开始时自动读取
- **持久记忆：** 在该目录的所有对话中保留
- **团队可共享：** 提交到 git，整个团队获得相同的上下文

### 宪法 vs 立法层级

理解这个层级至关重要：

| 方面 | CLAUDE.md（宪法） | 用户提示（立法） |
|---|---|---|
| **优先级** | 最高 - 总是获胜 | 次要 - 必须遵守 |
| **持久性** | 永久保留 | 每个请求临时 |
| **范围** | 适用于所有会话 | 适用于当前任务 |
| **目的** | 不可变的项目规则 | 灵活的具体请求 |
| **示例** | 始终称之为 Workspace | 为深色模式创建 PRD |

**如果有冲突，CLAUDE.md 总是获胜。**

### 示例：术语覆盖

**CLAUDE.md 说：**

```
在 TaskFlow 文档中始终使用 Workspace 而不是 Project。
```

**你提示：**

```
为新 Project 仪表板功能编写 PRD
```

**Claude 会：**

```
使用"Workspace 仪表板"编写 PRD，因为 CLAUDE.md 覆盖
你随意的提示措辞
```

## # 符号的力量

### 动态会话规则

使用行首的 **#** 为当前会话添加临时规则：

```
# 在此会话中始终使用项目符号而不是编号列表
```

Claude 将此视为对话其余部分的规则（但不是未来会话）。

### # vs CLAUDE.md

| 功能 | # 符号 | CLAUDE.md |
|---|---|---|
| **持续时间** | 仅当前会话 | 永久 |
| **范围** | 此对话 | 所有对话 |
| **用例** | 实验偏好 | 永久项目规则 |

**工作流程：** 使用 # 实验，然后当你想要永久化时添加到 CLAUDE.md。

## 📝 在 CLAUDE.md 中放什么

### ✅ 适合 CLAUDE.md

**1. 产品上下文**

```
## 产品概述

TaskFlow 是面向远程优先团队的项目管理 SaaS。
可以理解为 Asana 与 Jira 的结合，专为异步协作打造。

阶段：B 轮（已筹集 $20M，50 名员工）
收入：$2.5M ARR，10,000 活跃用户
```

**2. 用户画像**

```
## 用户画像

### Sarah - 企业管理员
- 角色：500 人以上公司的 IT 管理员
- 痛点：需要 SSO、审计日志、合规、安全
- 引用："我需要证明这符合我们的安全标准"
```

**3. 写作风格**

```
## 写作风格

- 使用主动语态（不是被动）
- 在所有列表中使用牛津逗号
- 最多 2 句段落以提高可读性
- 在文档中使用"我们"而不是"我"
```

**4. 产品术语**

```
## 产品术语

始终使用这些术语（绝不使用替代词）：
- Workspace 不是 Project - 我们的顶层容器
- Task 不是 Todo 或 Issue - 单个工作项
- PM 指产品经理（不是项目经理）
```

**5. 不可变规则**

```
## 不可变规则

这些规则覆盖任何冲突的提示：

1. 始终在用户故事中包含验收标准
 - 使用 Given/When/Then 格式
 - 使其具体且可测试

2. 绝不在没有用户研究引用的情况下编写 PRD
```

### ❌ 不要放在 CLAUDE.md

**1. 临时指令**

```
今天的会议笔记在 meeting-2025-10-13.txt
我们这个 sprint 正在做深色模式
```

*为什么不好：* 这些经常变化。改用提示。

**2. 频繁变化的需求**

```
当前 sprint 目标：实现 SSO
Q4 OKR：将激活率提高到 55%
```

*为什么不好：* 如果每周或每月变化，就不应该放在这里。

**3. 敏感信息**

```
我们的收入实际上在下降（不要告诉董事会）
API 密钥：sk-proj-xxxxx
```

*为什么不好：* CLAUDE.md 经常提交到 git 并共享。

### 测试

**如果你希望 Claude 在 6 个月后知道这个，放在 CLAUDE.md。**

**如果下周可能变化，改用提示。**

## 🏗️ CLAUDE.md 层级

### 四个级别

```
~/.claude/CLAUDE.md # 1. 全局（你所有项目）
/project-root/CLAUDE.md # 2. 项目特定
/project-root/frontend/CLAUDE.md # 3. 目录特定
/project-root/CLAUDE.local.md # 4. 个人（gitignored）
```

### 优先级顺序

**最具体的获胜：**

1. **目录级别**（如 `/frontend/CLAUDE.md`）

2. **项目级别**（如 `/project-root/CLAUDE.md`）

3. **全局**（如 `~/.claude/CLAUDE.md`）

4. **用户提示**（最低优先级）

**它们如何叠加：** 级别组合（不替换）。所有规则同时应用，更具体的级别在冲突时覆盖。

### 何时使用每个级别

| 级别 | 用于 | 示例 |
|---|---|---|
| **全局** | 所有工作的个人偏好 | "我偏好简短摘要，最多 3 点" |
| **项目** | 每个人都需要的产品特定上下文 | 产品概述、画像、术语 |
| **目录** | 项目子部分的规则 | Frontend：移动响应式，Backend：API 文档 |
| **个人** | 你不想分享的偏好 | 个人响应格式偏好 |

**重要：** 将 `CLAUDE.local.md` 添加到 `.gitignore`：

```
# .gitignore
CLAUDE.local.md
```

## 🚀 创建你的第一个 CLAUDE.md

### 快速入门模板

```

此文件为 Claude Code 提供关于 [产品] 的永久上下文。

**公司详情：**
- 成立时间：[年份]
- 阶段：[阶段和融资]
- 团队：[规模]
- 收入：[ARR 和用户]

## 用户画像

- 痛点：[什么让他们沮丧]
- 引用：[令人难忘的引用]

## 写作风格与标准

- 语态：[主动/被动，正式/随意]
- 段落长度：[偏好]
- 语气：[应该听起来如何]

## 产品术语

始终使用这些术语（绝不使用替代词）：
- [术语] 不是 [替代词] - [原因]
- [术语] 不是 [替代词] - [原因]

## 不可变规则

1. 始终 [规则]

```

### 示例：TaskFlow CLAUDE.md

```
# TaskFlow 产品上下文

此文件为 Claude Code 提供关于 TaskFlow 的永久上下文。

## TaskFlow 是什么

TaskFlow 是面向远程优先团队的项目管理 SaaS。可以理解为 Asana
与 Jira 的结合，但专为异步协作打造。

**公司详情：**
- 成立时间：2021
- 阶段：B 轮（已筹集 $20M）
- 团队：50 名员工
- 收入：$2.5M ARR，10,000 活跃用户

**你的角色：**
- 职位：高级产品经理
- 关注：激活与引导

## 用户画像

### Sarah - 企业管理员
- 角色：500 人以上公司的 IT 管理员
- 痛点：需要安全、SSO、审计日志、合规
- 引用："我需要证明 TaskFlow 符合我们的安全标准"

### Mike - IC 工程师
- 角色：创业公司的个人贡献者软件工程师
- 痛点：想要速度、键盘快捷键、GitHub 集成
- 引用："如果我必须使用鼠标，这个工具就太慢了"

## 写作风格与标准

- 使用主动语态（不是被动）
- 在所有列表中使用牛津逗号
- 最多 2 句段落以提高可读性
- 在文档中使用"我们"而不是"我"

## 产品术语

始终一致使用这些术语：
- Workspace 不是 Project - 我们的顶层容器
- Task 不是 Todo 或 Issue - 单个工作项
- Epic 不是 Initiative - 大型多 sprint 功能

## 不可变规则

1. 始终在用户故事中包含验收标准
 - 使用 Given/When/Then 格式
 - 使其具体且可测试

2. 绝不在没有用户研究引用的情况下编写 PRD
 - 链接到访谈、调查或支持工单

## 团队参考

- Sarah - CEO，创始人，愿景和融资
- Mike - CTO，技术架构和工程
- Alex - 设计主管，负责所有 UX 和视觉设计
- 你 - 高级 PM，激活与引导
```

## 💡 最佳实践

**要具体，不要模糊：**

- ❌ "用户喜欢简单的界面"

- ✅ "用户研究（8/10 访谈，2025 年 6 月）显示用户放弃超过 3 个必填字段的功能。保持表单最小化。"

**使用祈使语气：**

- ❌ "如果用户故事有验收标准就好了"

- ✅ "始终在用户故事中包含验收标准"

**提供上下文：**

- ❌ "使用 Workspace 不是 Project"

- ✅ "使用 Workspace 不是 Project - 我们与传统项目管理工具区分开来。Workspace 表示协作空间。"

**保持可扫描：**

- 大量使用项目符号
- 短段落（1-2 句）
- 加粗关键术语
- 在部分之间添加间距

**定期维护：**

- 每季度审查（画像、术语、指标）
- 提交到 git 并在 PR 中审查变更
- 保持最新（目标是 50-200 行的最佳范围）

## 🔧 故障排除

**规则被忽略？**

- 检查层级级别之间的冲突规则
- 使规则明确：使用 ALWAYS/NEVER
- 要具体，不要模糊

**CLAUDE.md 没有加载？**

- 验证文件存在：`ls CLAUDE.md`
- 必须完全命名为 `CLAUDE.md`（区分大小写）
- 检查文件权限：`ls -la CLAUDE.md`
- 测试：问"我的 CLAUDE.md 说什么？"

**CLAUDE.md 文件太多？**

- 审计所有：`find . -name CLAUDE.md`
- 在 README 中记录层级
- 使用 CLAUDE.local.md 存储个人偏好

**CLAUDE.md 太长？**

- 目标是 50-200 行
- 将详细上下文移到单独的参考文档
- 从 CLAUDE.md 链接到这些文档
