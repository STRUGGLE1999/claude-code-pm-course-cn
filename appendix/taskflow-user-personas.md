# TaskFlow 用户画像

**深入了解谁在使用 TaskFlow 以及为什么**

## 概述

TaskFlow 在客户群中服务三类主要用户画像。理解这些画像对产品决策至关重要——每个功能都应该至少服务一个画像的核心需求。

## 画像1：Sarah（企业管理员）

### 快速信息

**姓名：** Sarah Chen **年龄：** 38岁 **角色：** IT 运营副总裁 **公司：** TechCorp（800员工，C轮） **地点：** 德克萨斯州奥斯汀（远程团队遍布美国） **TaskFlow 计划：** 企业版

### 背景

Sarah 领导一家快速增长的 SaaS 公司的 IT 运营。她的团队支持800名员工使用15个工具。她向 CTO 汇报，负责 IT 预算（每年$2M）。

**职业路径：**

- 从 IT 支持专员开始

- 成长为 IT 经理

- 现任 IT 运营副总裁（6名直接下属）

- IT 行业12年，在现公司8年

**教育背景：**

- 信息系统学士学位

- CISSP 认证（安全）

### 目标与动机

**主要目标：**

1. **减少工具分散** - 目前使用25+工具，希望整合到10-12个

2. **改善安全态势** - 通过 SOC 2 审计，满足企业客户要求

3. **控制成本** - $2M 预算，需要展示每个工具的 ROI

4. **赋能团队** - 给员工好工具，不妥协安全

**成功指标：**

- 使用中的工具数量（下降）

- 安全事件（零）

- 员工对工具的满意度（上升）

- 每员工工具成本（下降）

### 痛点与挫折

**1. 工具分散失控**

- 工程团队使用 Linear

- 市场团队使用 Monday.com

- 销售团队使用 Salesforce + Asana

- 设计团队使用 Notion

- 没有中心位置查看正在发生什么

**2. 安全噩梦**

- 影子 IT（团队未经批准购买工具）

- 无法查看谁有权访问什么

- 离职员工仍可访问工具

- 被要求时无法提供审计日志

**3. 入职/离职混乱**

- 新员工？需要手动创建25个账户

- 有人离开？需要撤销25个访问权限

- 完全配置一个人需要2天

**4. 没有可见性**

- 无法看到团队在做什么

- 无法向高管汇报

- 无法预测资源需求

### 待办任务

**当……**

- ……新员工入职 → 在一个地方自动配置所有工具访问权限

- ……有人离开 → 立即确信地撤销所有访问权限

- ……审计员要求日志 → 无需慌乱地提供全面的审计追踪

- ……评估新工具 → 购买前确保符合安全标准

- ……CFO 询问工具成本 → 展示每个工具的 ROI 和使用指标

- ……发生安全事件 → 立即识别访问范围并撤销

### 他们关心的 TaskFlow 功能

**必须有：**

- ✅ 通过 SAML 的 SSO（单点登录）- 不可协商

- ✅ 审计日志 - 谁在何时、为什么做了什么

- ✅ 高级权限 - 基于角色的访问控制

- ✅ 批量用户管理 - 一次添加/删除多个用户

- ✅ 管理员仪表板 - 组织级可见性

**最好有：**

- 自定义用户字段（部门、地点、成本中心）

- 使用分析（谁在用，谁没用）

- 数据导出（拥有你的数据）

- 99.9% 正常运行时间 SLA

**不关心：**

- 花哨的 UI 动画

- 社交功能

- 公开看板

- 移动应用（桌面优先角色）

### 行为与偏好

**工作方式：**

- 深度专注：大块的专注工作（无持续干扰）

- 沟通：邮件 > Slack（偏好异步）

- 决策：数据驱动，风险规避

- 工具：喜欢仪表板、报告、分析

**技术熟练度：** 高

- 对技术概念感到舒适

- 阅读安全白皮书

- 严格评估工具

**购买流程：**

- 做大量研究（数周）

- 需要安全审查

- 需要法务审查合同

- 先用50个用户试点再全面推广

- 典型销售周期：3-6个月

### 语录

*我需要一个地方可以看到谁有权访问什么，并在需要时立即撤销。*

*安全不可协商。如果你没有 SSO 和审计日志，我们甚至无法考虑你。*

*我厌倦了成为每个工具评估的瓶颈。我需要那些能够正常工作并符合我们标准的工具。*

*当员工离开时，我会失眠担心我们是否已撤销他们所有的访问权限。不应该这么难。*

## 画像2：Mike（IC 工程师）

### 快速信息

**姓名：** Mike Rodriguez **年龄：** 29岁 **角色：** 高级软件工程师 **公司：** GrowthLabs（150员工，B轮） **地点：** 俄勒冈州波特兰（完全远程） **TaskFlow 计划：** 专业版（公司范围）

### 背景

Mike 是专注于后端系统的高级工程师。他每天编码6-8小时，每周参加2-3次会议，与分布式团队跨4个时区工作。

**职业路径：**

- 编程训练营毕业（2018年）

- 初级 → 中级 → 高级工程师（5年）

- 专长：Node.js、PostgreSQL、AWS

**教育背景：**

- 编程训练营（Hack Reactor）

- 自学（训练营之前）

### 目标与动机

**主要目标：**

1. **深度工作** - 长时间、无干扰的专注时间来解决复杂问题

2. **清晰优先级** - 准确知道做什么，不猜测

3. **最小化上下文切换** - 更少工具，更少跳转

4. **交付高质量代码** - 对工匠精神自豪

**成功指标：**

- 每冲刺发布的功能

- Bug 数量（低）

- 代码审查周转时间（快）

- 专注时间（每天6+小时）

### 痛点与挫折

**1. 优先级混乱**

- 多个事实来源（Slack、邮件、站会、任务工具）

- 来自不同利益相关者的冲突优先级

- 不清楚"紧急"到底意味着什么

- 浪费时间问"我应该做什么？"

**2. 上下文不足**

- 任务标题："修复登录 bug"（哪个 bug？在哪里？为什么？）

- 需求中没有技术细节

- 产品规格缺乏边界情况

- 必须寻找背景信息

**3. 会议过载**

- 站会可以异步

- 状态更新打断专注

- 会议安排在最佳专注时间

- 时区让调度更糟

**4. 缓慢、臃肿的工具**

- 页面加载时间 > 3秒

- 沉重、卡顿的界面

- 做简单事情需要太多点击

- 移动网页无法使用

### 待办任务

**当……**

- ……开始工作 → 立即看到最高优先级任务，附完整上下文

- ……被某事阻塞 → 无需安排会议即可沟通阻碍

- ……完成任务 → 无需询问就知道下一步是什么

- ……审查 PR → 无需离开 GitHub 即可看到链接的任务上下文

- ……规划容量 → 看到即将到来的工作并估算工作量

### 他们关心的 TaskFlow 功能

**必须有：**

- ✅ 键盘快捷键 - 无需鼠标导航（Cmd+K、j/k 导航）

- ✅ 快速性能 - 亚秒级页面加载，无转圈

- ✅ GitHub 集成 - PR 链接到任务，自动状态更新

- ✅ 丰富的 markdown - 代码块、语法高亮

- ✅ 任务上下文 - 清楚的为什么，不只是什么

**最好有：**

- 深色模式（深夜编码）

- 离线模式（飞机上工作）

- API 访问（自动化）

- CLI 工具

**不关心：**

- 可视化项目看板（偏好列表视图）

- 图表和图形

- 社交功能

- 视频通话

### 行为与偏好

**工作方式：**

- 深度专注：2-3小时无干扰编码块

- 沟通：异步 > 实时（专注时关闭 Slack）

- 决策：技术价值优先，效率

- 工具：键盘 > 鼠标，CLI > GUI

**技术熟练度：** 非常高

- 开发者工具的高级用户

- 编写脚本自动化工作流程

- 定制一切

**工作时间：**

- 灵活（远程）

- 最佳生产力：上午（6am-12pm）

- 避免中午前的会议

- 如果早些时候被阻塞，经常在晚上工作

### 语录

*只要告诉我构建什么，给我上下文，让我编码。不要让我寻找信息或坐在会议里。*

*如果你的工具慢，我就不会用它。我会找到更快的替代品或自己构建。*

*我需要所有东西都有键盘快捷键。如果我必须使用鼠标，我已经恼火了。*

*最好的站会是那些不发生的站会。让我异步阅读更新就好。*

## 画像3：Alex（团队负责人）

### 快速信息

**姓名：** Alex Rivera **年龄：** 35岁 **角色：** 工程经理 **公司：** DataFlow（200员工，B轮） **地点：** 加利福尼亚州旧金山（远程团队遍布5个时区） **TaskFlow 计划：** 专业版 **团队规模：** 8名工程师（3名高级，5名中级）

### 背景

Alex 管理一个分布式工程团队。前高级工程师，2年前晋升为经理。仍偶尔编码（约20%时间），但主要专注于团队生产力、招聘和交付。

**职业路径：**

- 工程师（5年）

- 技术负责人（2年）

- 工程经理（2年）

- 目标：工程总监

**教育背景：**

- 计算机科学学士学位

- MBA（晚间课程，进行中）

### 目标与动机

**主要目标：**

1. **团队成功** - 团队按时、高质量交付，无倦怠

2. **清晰可见性** - 无需询问即可知道状态，尽早识别阻碍

3. **平衡的工作负载** - 无人超负荷，无人利用不足

4. **职业成长** - 发展团队成员，招聘 A 级人才，成长为总监

**成功指标：**

- 冲刺速度（一致、可预测）

- 团队幸福感（留存、参与度）

- 按时交付率

- 代码质量（低 bug 率）

### 痛点与挫折

**1. 不询问就没有可见性**

- 必须联系团队成员："你的状态是什么？"

- 站会是状态更新（应该是异步）

- 当阻碍浮出水面时，它们已经是关键问题

- 直到太晚才能看到工作负载平衡

**2. 持续的上下文切换**

- 检查5个工具来了解状态

- GitHub（代码）、TaskFlow（任务）、Slack（沟通）、Figma（设计）、Notion（文档）

- 难以获得全局图景

- 浪费时间聚合信息

**3. 向上汇报痛苦**

- 领导层问："我们按计划吗？"

- 需要2小时收集数据并创建更新

- 当我汇报时，数据已经过时

- 应该自动化的手工工作

**4. 团队平衡问题**

- 一些工程师超负荷（晚上工作）

- 其他人利用不足（等待任务）

- 难以一目了然看到容量

- 任务分配是猜测工作

### 待办任务

**当……**

- ……规划冲刺 → 看到团队容量，公平分配工作

- ……有人被阻塞 → 无需每日站会即可识别阻碍

- ……向领导层汇报 → 清晰快速地展示团队进展

- ……与工程师1:1 → 在上下文中看到他们的工作，提供反馈

- ……出现招聘需求 → 向领导层展示我们已达容量（需要人头）

### 他们关心的 TaskFlow 功能

**必须有：**

- ✅ 团队仪表板 - 一目了然看到每个人的任务

- ✅ 工作负载视图 - 谁超负荷，谁有容量

- ✅ 阻塞任务可见性 - 红色标记，紧急关注

- ✅ 冲刺报告 - 速度、燃尽、可预测性

- ✅ 评论摘要 - 无需阅读所有内容即可跟进讨论

**最好有：**

- 预测分析（按当前速度团队将延迟2天完成）

- 历史速度（比较冲刺）

- 个人表现追踪（用于评估）

- 与日历集成（看到谁外出、谁有空）

**不关心：**

- 细粒度任务细节（信任团队处理）

- 花哨的动画

- 社交功能

### 行为与偏好

**工作方式：**

- 管理时间：会议、1:1、规划（50%）

- 编码时间：偶尔的 IC 工作（20%）

- 战略时间：招聘、路线图、团队发展（30%）

- 沟通：同步（1:1）和异步（更新）混合

**技术熟练度：** 高

- 前工程师（理解技术细节）

- 对数据和分析感到舒适

- 欣赏自动化

**工作时间：**

- 灵活（远程）

- 会议：9am-5pm

- 编码：清晨或深夜（无干扰）

- 可用性：跨时区（一些晚上的电话）

### 语录

*我需要知道我的团队是否按计划进行，而不必每天单独询问他们。那不可扩展。*

*当我整天在连续会议中时，我需要在5分钟内跟进团队进展，而不是2小时。*

*我信任我的团队，但我需要可见性。如果有人被阻塞而我不知道，那是我的责任。*

*向上汇报不应该是一个手工过程。数据就在那里——只需以一种我可以分享的格式展示给我。*

## 画像如何影响产品决策

### 示例：深色模式功能

**Sarah 的视角：**

- 不关心（桌面优先角色，正常时间工作）

- 不是阻碍，也不是驱动因素

**Mike 的视角：**

- **喜欢它**（深夜编码，减少眼睛疲劳）

- 会因为这个积极推荐 TaskFlow

- 与竞争对手的差异化因素

**Alex 的视角：**

- 最好有（一些工程师工作到很晚）

- 对团队成功不关键

- 会让团队开心

**决策：** 发布深色模式

- 对 Mike 画像价值高（留存 + 推荐）

- 构建成本低

- 向市场传递积极信号（现代工具）

### 示例：高级权限

**Sarah 的视角：**

- **必须有**（没有这个无法购买）

- 企业交易的阻碍

- 愿意支付溢价

**Mike 的视角：**

- 不关心（不是他们的工作）

- 只要不拖慢工具就行

**Alex 的视角：**

- 有用（控制团队看到什么）

- 不关键

**决策：** 发布高级权限

- 企业细分市场的阻碍（高收入）

- 基本功能

- 向上拓展扩展所需

## 画像驱动的路线图优先级

**2025年第一季度优先级：**

1. **移动应用** → 所有画像受益

    - Sarah：团队在移动中使用

    - Mike：通勤时审查任务

    - Alex：旅行时检查团队状态

2. **SSO 与企业功能** → Sarah 的阻碍

    - 企业细分市场关键

    - 高收入影响

    - 竞争要求

3. **激活改进** → Mike 的首次体验

    - 更快的价值实现

    - 更好的引导

    - 减少流失

4. **深色模式** → Mike 的愉悦因素

    - 高度请求

    - 差异化

    - 工程团队士气
